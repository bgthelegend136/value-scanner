## Main 2-API Telegram scanner

Decision:

- Keep the strict 2-API scanner for Stoiximan/Pamestoixima.
- Keep World Cup active through `config/multisport-map.json`.
- Lower/keep the confirmed Telegram floor at 5% EV, not 10%.
- Do not send Telegram for paper-only rows, totals research rows, unsupported
  markets, stale candidates, or rows that fail Pinnacle/consensus confirmation.

Telegram alert criteria:

1. Odds-API.io finds a Stoiximan or Pamestoixima candidate.
2. Candidate market is full-event `MATCH_RESULT` / h2h / 1X2.
3. Candidate provider EV is at least 5%.
4. The event is pre-match and the quote is fresh.
5. The league maps cleanly to an active The Odds API sport key.
6. The Odds API can match the same event and exact selection.
7. Pinnacle de-vigged EV is strictly above 5%.
8. Median consensus EV from at least three complete reference books is strictly
   above 5%.
9. Candidate has not already been sent.
10. Telegram delivery succeeds.

Observed result:

- `reports/mispricing-alerts.csv` contains two sent World Cup Stoiximan alerts
  from 2026-06-26.
- `telegram-test` was run on 2026-06-29 and returned `message 9`; the user
  confirmed receiving it. Telegram delivery is therefore working.
- Recent quiet periods are mainly because the scanner did not find new clean
  confirmed +5% h2h opportunities, not because Telegram was broken.

## Betsson one-API POC

Decision:

- Keep the Betsson single-provider experiment isolated in the
  `codex/betsson-single-api-poc` worktree.
- Use The Odds API only, with Betsson as the candidate book and other books on
  the same event id as consensus.
- Include World Cup in the scheduled scan.
- Enable Telegram only as h2h watchlist when `--telegram-watchlist` is present.
- Keep `TOTALS`, `BTTS`, `DRAW_NO_BET`, and `DOUBLE_CHANCE` research-only.

Reason:

- One API removes cross-provider event matching errors.
- It is cleaner software architecture, but it only covers books available in
  The Odds API and still needs separate Betsson CLV/ROI evidence.
